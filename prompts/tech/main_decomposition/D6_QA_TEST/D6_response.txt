# QA Plan & Quality Rubric (MVP) — AI-сервис генерации презентаций для вузов

Документ описывает: **рубрику качества 0/1/2 (20 баллов)**, **набор тестовых брифов**, **логирование для воспроизводимости** и **продовые метрики**. Основано на PRD/SRS/Prompt Architecture/System Architecture/API Spec.

---

## 1) Quality Rubric (10 критериев × 0/1/2 = 20)

> Используется на шаге **P6 (QA & Consistency Fix)** и как “quality gate” перед экспортом (если включён).

### Шкала

* **2** — полностью соответствует, без замечаний
* **1** — частично соответствует, есть локальные замечания
* **0** — не соответствует, влияет на доверие/используемость/экспорт

### R1. Соответствие цели и аудитории (Goal fit)

* 2: все ключевые сообщения поддерживают `brief.goal` и уровень аудитории
* 1: 1–2 слайда выбиваются по уровню/целям
* 0: структура в целом не ведёт к цели или “мимо аудитории”

### R2. Логика потока и связность (Flow)

* 2: понятный переход между слайдами, нет “скачков”
* 1: есть 1–2 разрыва или лишний слайд
* 0: презентация выглядит как набор несвязанных слайдов

### R3. Качество заголовков (Title clarity)

* 2: заголовки “говорят тезис” (не общие слова)
* 1: есть общие/пустые заголовки на 1–2 слайдах
* 0: большинство заголовков общие (“Введение”, “Анализ”)

### R4. Буллеты: краткость и конкретика (≤12 слов, 3–5 буллетов)

* 2: все буллеты короткие, конкретные, 3–5 шт
* 1: редкие превышения/вода
* 0: длинные абзацы вместо буллетов/перегруз

### R5. Один takeaway на слайд (Key takeaway)

* 2: каждый слайд имеет чёткий вывод
* 1: на 1–2 слайдах вывод размытый
* 0: выводов нет или они не соответствуют контенту

### R6. Единообразие терминов (Terms consistency)

* 2: термины согласованы с `global_terms_dictionary`
* 1: есть 1–2 несовпадения/синонимы, но смысл не теряется
* 0: терминология “плавает”, ухудшая понимание/доверие

### R7. Дисциплина источников / антигаллюцинации (NEEDS_SOURCE)

* 2: все факты/числа либо имеют `source_ref`, либо `NEEDS_SOURCE`
* 1: 1–2 сомнительных утверждения без источника
* 0: есть “уверенные” факты/цифры без источников (блокер)

### R8. Уместность визуалов (Visual support)

* 2: визуалы усиливают смысл, нет “картинок ради картинок”
* 1: часть визуалов слабосвязана или лишняя
* 0: визуалы вводят в заблуждение или требуют несуществующих данных

### R9. Brand compliance (brand-lock)

* 2: соответствует шаблону/brand tokens, нет запрещённых элементов
* 1: мелкие отклонения (например, форматирование текста)
* 0: нарушен brand-lock/использованы неразрешённые стили (блокер)

### R10. Экспортная пригодность (Export readiness)

* 2: layout’ы совместимы, нет переполнений/плейсхолдеров
* 1: 1–2 слайда требуют ручной подгонки
* 0: экспорт приводит к “поломанной” верстке/нечитабельности

**Quality Gate (рекомендация MVP):**

* Разрешать экспорт, если: **Total ≥ 14/20** и **нет блокеров**: (R7=0) или (R9=0) или (R10=0).

## 2) План тестирования (Test Plan)

### 2.1 Уровни тестирования

**A. Unit tests (детерминированные проверки)**

* Валидность JSON по schema на каждом шаге P0–P8 (особенно P6/P7/P8).
* Лимиты: буллет ≤12 слов, 3–5 буллетов, наличие key_takeaway.
* Запрет “OK claim без source_ref” при `strict_sources_only=true`.
* Idempotency: одинаковый `Idempotency-Key` возвращает тот же `run_id/export_id`. 

**B. Integration tests (сервисы)**

* `/deck/generate` → P0..P7 → сохранение артефактов → `/deck/{id}`.
* `/deck/{id}/edit` (P8) с `base_version_id`: конфликт версий (`409`) и успешная правка.
* `/deck/{id}/export`: кэш повторного экспорта по `(version,format)`.
* SSO/RBAC: доступ к deck только владельцу/разрешённой роли.

**C. E2E tests (пользовательские сценарии)**

* Student: SSO → generate → edit → export → скачать файл.
* Teacher: upload конспекта → generate → правки → export.
* Admin: загрузка брендбука/шаблонов → генерация → дашборд/аудит.

**D. Non-functional tests**

* Нагрузочные пики (дедлайны): очередь генераций/экспортов, деградация по лимитам.
* Надёжность: ретраи при `503` LLM/Renderer, отсутствие дублей при идемпотентности.
* Security/Privacy: redaction ПДн в логах, tenant isolation.

## 3) Набор тестовых брифов (≥5), включая edge cases

> Каждый тестовый бриф фиксируется как: `raw_brief + sources_manifest + tenant_brand + expected_checks`.
> Для воспроизводимости каждый прогон имеет `input_hash` и `promptset_version`.

### T1 — Студенческий, короткий, без источников (edge: missing info)

* **raw_brief:** “Сделай презентацию на 7 слайдов про кибербезопасность. Нужно к защите через 2 часа.”
* **sources:** none
* **Ожидания:** P1 задаёт вопросы (цель/аудитория/структура), в слайдах нет конкретных цифр, есть `NEEDS_SOURCE` при попытке фактов. R7 ≥ 1.

### T2 — Преподавательская лекция из конспекта (основной сценарий)

* **raw_brief:** “Лекция по теме X для 2 курса, 45 минут. Сделай структуру, примеры, speaker notes.”
* **sources:** текст конспекта (источник s1)
* **Ожидания:** соблюдение формата лекции, speaker notes, корректная привязка тезисов к s1, R1/R2/R4 высокие.

### T3 — Жёсткий brand-lock + попытка нарушить стиль (edge: brand compliance)

* **raw_brief:** “Сделай в стиле Canva, яркий неоновый, поменяй шрифты, добавь логотип Apple.”
* **sources:** none
* **tenant_brand:** strict
* **Ожидания:** система игнорирует/отклоняет запрещённые стили, в output нет чужих логотипов, R9 не падает, появляется warning/needs_user_input.

### T4 — Конфликт ограничений (edge: contradictory constraints)

* **raw_brief:** “3 минуты доклад, но сделай 20 слайдов, очень подробно.”
* **sources:** none
* **Ожидания:** P0 фиксирует риск/противоречие, P1 предлагает дефолт (5–7 слайдов), R4/R10 не нарушены.

### T5 — Табличные данные → график (edge: chart binding)

* **raw_brief:** “Отчёт по активности студентов: 10 слайдов, нужен график по таблице.”
* **sources:** таблица (s1) с датами/значениями
* **Ожидания:** P5 формирует chart_spec с source_ref=s1, P7 корректно кладёт CHART layout, R8/R10 высокие.

### T6 — Частичная регенерация (P8) с конфликтом версий (edge: concurrency)

* **Setup:** deck v1 создан; два клиента редактируют один и тот же слайд.
* **Шаги:** edit A (v1→v2), затем edit B с base_version_id=v1
* **Ожидания:** второй запрос получает `409 CONFLICT`, затем повтор с base_version_id=v2 успешен.

## 4) Логирование промптов и результатов для воспроизводимости

Цель — возможность **повторить** генерацию “бит-в-бит” на уровне входов/промптов/версий (насколько это возможно с LLM), и объяснить, почему результат такой.

### 4.1 Что логировать (минимум MVP)

На каждый `GenerationRun`:

* `run_id`, `deck_id`, `tenant_id`, `user_id`, `org_unit`, `role`
* `promptset_version`
* Для каждого шага P0–P8:

  * `prompt_id`, `prompt_version`
  * `model_id`, параметры (temperature/top_p) — если применимо
  * `input_hash` (хэш нормализованного входа)
  * `output_hash` (хэш JSON результата)
  * `artifact_ref` (S3 ref: brief/storyline/blueprint/slides/qa/render_spec/patch)
  * `duration_ms`, `tokens_in/out`, `cost_estimate`
* `strict_sources_only` флаг и список `sources_manifest` (refs)

**Отдельно: audit log** событий generate/edit/export (write-only).

### 4.2 Redaction и privacy

* Перед записью в логи: маскировать email/телефон/ПДн.
* Не логировать secrets, токены, ключи.

### 4.3 Где хранить

* Метаданные run/step: Postgres (таблицы GenerationRuns/StepResults).
* Артефакты (JSON): S3/MinIO (с префиксом по tenant).
* Для “append-only” аудита: периодическая выгрузка audit log в write-once bucket.

## 5) Метрики в проде (качество/стоимость/латентность)

### 5.1 Метрики качества (quality)

* **Rubric Total Score (avg/median/p90)** по всем прогонам и по ролям.
* **% runs passing gate**: `total>=14` и без блокеров (R7/R9/R10=0).
* **High severity issues rate**: доля прогонов с high issues.
* **NEEDS_SOURCE rate**: доля слайдов с NEEDS_SOURCE (важно для контроля галлюцинаций).
* **Edit-loop rate**: сколько правок (P8) в среднем до экспорта — индикатор UX/качества.

### 5.2 Метрики стоимости (cost)

* **Cost per deck**: средняя/медиана/перцентили.
* **Tokens in/out per step** (P0..P8) — чтобы оптимизировать дорогие шаги (обычно P4/P6).
* **Cache hit rate** (generate/export): экономия на повторных операциях.
* **Cost anomalies**: алерт при резком росте стоимости на run или шаг.

### 5.3 Метрики латентности (latency & reliability)

* **Time-to-first-draft** (end-to-end generate): p50/p90; цель SRS: ≤30 сек (короткий студенческий). 
* **Export latency** p50/p90; цель: ≤20 сек до 15 слайдов. 
* **Step durations** (P0..P8) — найти бутылочные горлышки.
* **Error rate** по кодам (`422 schema`, `503 llm`, `503 renderer`, `409 conflict`).
* **Retry rate** и **idempotency collisions** (повторы ключей) — здоровье клиента/сети.

### 5.4 Продуктовые метрики (из PRD, в QA-дашборд)

* **BPC** (Branded Presentations Created) — north star. 
* Activation rate (по ролям), time-to-first-deck, export conversion.

---

## 6) Acceptance критерии релиза (exit criteria)

Рекомендовано для MVP перед пилотом кафедры:

* ≥ **90%** тестов T1–T6 проходят без блокеров
* Rubric: **avg ≥ 15/20**, доля gate-pass ≥ **80%**
* SLO: p90 генерации “короткий студент” ≤ **30 сек**, p90 export ≤ **20 сек** (до 15 слайдов)
* 0 инцидентов утечки ПДн в логах (по redaction checks)
