<<<{# SRS: AI-сервис генерации презентаций для российских вузов (B2B2C)

> Документ описывает требования к MVP сервиса “Brand-Locked AI” для создания академических презентаций в стиле вуза, с управляемостью для закупщика и удобством для преподавателей/студентов.

---

## 1) Сущности и термины (глоссарий)

**Вуз (University / Tenant)** — организация-клиент; логическая изоляция данных и настроек.
**Пользователь (User)** — человек с ролью: *Student / Teacher / Admin / Buyer* (Buyer может совпадать с Admin).
**Роль (Role)** — набор прав в системе (RBAC).
**Кафедра/Факультет (Org Unit)** — организационная единица внутри вуза; используется для шаблонов и отчётности.
**Бриф (Brief)** — входные требования пользователя: тема, цель, аудитория, время доклада, стиль, ограничения, источники.
**Источник (Source)** — документ/текст/ссылка/таблица, из которого разрешено брать факты.
**Презентация (Deck)** — сущность результата; включает структуру, слайды, заметки, историю изменений.
**Слайд (Slide)** — элемент Deck: заголовок, буллеты, визуалы, заметки докладчика.
**Шаблон (Template)** — макеты слайдов и правила оформления (layout + стили).
**Брендбук (Brand Guidelines)** — набор токенов: цвета/шрифты/логотип/правила применения.
**Brand-lock** — механизм, запрещающий пользователю “сломать” ключевые параметры брендбука (например: шрифты/цвета/логотипы).
**Генерация (Generation Run)** — один запуск конвейера: вход → план → слайды → QA → экспорт.
**Экспорт (Export)** — выгрузка результата в PPTX/PDF (и/или Google Slides).
**Метрики (Metrics)** — измерения использования: BPC, activation rate, time-to-first-deck, и т.п.
**PII/ПДн** — персональные данные; любые данные, подпадающие под требования локального комплаенса.
**SLO** — целевой уровень сервиса (латентность/доступность).

## 2) User stories (минимум 10)

### 2.1 Для студента (Student)

1. **As a Student, I want to sign in via SSO, so that I can start without registration friction.**
2. **As a Student, I want to generate a deck from a topic or text, so that I can meet a deadline quickly.**
3. **As a Student, I want a progress indicator and ETA, so that I understand when I will get the result.**
4. **As a Student, I want to export to PPTX/PDF, so that the file opens reliably during the defense.**
5. **As a Student, I want to request “simplify / shorten / add slide”, so that I can iterate without rewriting everything.**

### 2.2 Для преподавателя (Teacher)

6. **As a Teacher, I want to upload a lecture note (text/doc), so that I get a structured lecture deck fast.**
7. **As a Teacher, I want speaker notes generated per slide, so that I can deliver consistently.**
8. **As a Teacher, I want to reuse and update a deck next semester, so that I don’t rebuild slides from scratch.**
9. **As a Teacher, I want references/traceability to sources, so that I trust the generated content and avoid errors.**

### 2.3 Для администратора вуза (Admin)

10. **As an Admin, I want to upload brand guidelines and templates, so that all decks are automatically branded.**
11. **As an Admin, I want RBAC and org units (faculties/departments), so that access and reporting are controlled.**
12. **As an Admin, I want dashboards by org unit (BPC, activation, exports), so that I can report digitalization metrics.**
13. **As an Admin, I want audit logs of generation and exports, so that I can investigate incidents and meet compliance needs.**

### 2.4 Для закупщика/руководства (Buyer)

14. **As a Buyer, I want a demo/pilot without barriers, so that I can evaluate value before procurement.**
15. **As a Buyer, I want a “document pack” (policy, DPA/ПДн, hosting confirmation), so that procurement is fast and low-risk.**

## 3) Acceptance criteria (Given/When/Then) для ключевых историй

### AC-1: Вход через SSO (User story #1)

* **Given** пользователь относится к вузу-тенанту и у вуза включен SSO,
  **When** пользователь нажимает “Войти”,
  **Then** он аутентифицируется через SSO-провайдер и попадает в рабочее пространство вуза без отдельной регистрации.
* **Given** SSO недоступен/ошибка,
  **When** попытка входа завершается сбоем,
  **Then** пользователь видит понятное сообщение и вариант повторить попытку/перейти на альтернативный способ входа (если разрешено политикой вуза).

### AC-2: Генерация презентации из текста (User story #2 / #6)

* **Given** пользователь заполнил brief (тема/цель/аудитория/язык) и загрузил текст/вставил контент,
  **When** он нажимает “Сгенерировать”,
  **Then** создаётся Generation Run, показывается прогресс, и по завершении появляется черновик deck с не менее чем N слайдами (N задаётся по умолчанию или пользователем).
* **Given** в brief отсутствуют критичные поля (например, цель или аудитория),
  **When** пользователь запускает генерацию,
  **Then** система задаёт до 7 уточняющих вопросов или применяет дефолты с пометкой “assumed” и предлагает подтвердить.

### AC-3: Brand-lock и применение шаблонов (User story #10)

* **Given** у тенанта загружены бренд-гайд и шаблон,
  **When** создается любой deck,
  **Then** все слайды применяют бренд-токены (цвета/шрифты/логотипы), а запрещённые изменения (brand-locked параметры) недоступны пользователю.
* **Given** пользователь пытается изменить brand-locked параметр,
  **When** он делает правку,
  **Then** система отклоняет изменение и предлагает допустимую альтернативу (например, выбрать из разрешённых макетов).

### AC-4: Экспорт PPTX/PDF (User story #4)

* **Given** deck успешно сгенерирован и прошёл минимальный QA,
  **When** пользователь нажимает “Экспорт в PPTX”,
  **Then** он получает файл, который открывается в MS PowerPoint без потери верстки (в пределах выбранных шаблонов) и содержит speaker notes (если включено).
* **Given** экспорт не удался (ошибка рендера/таймаут),
  **When** операция завершилась ошибкой,
  **Then** пользователь видит код ошибки, рекомендации (retry), а система сохраняет попытку в логах.

### AC-5: Итеративные правки “изменить слайд/сократить” (User story #5 / #8)

* **Given** пользователь выбрал конкретный слайд и команду правки (“сократи”, “упрости”, “добавь пример”),
  **When** он применяет изменение,
  **Then** перегенерируется только выбранный слайд (или блок), остальная презентация не меняется, и сохраняется история версий.

### AC-6: Дашборды и отчётность (User story #12)

* **Given** в системе есть события генерации/экспорта,
  **When** администратор открывает дашборд,
  **Then** он видит метрики (BPC, activation rate, exports, time-to-first-deck) в разрезе кафедр/факультетов и периода, с возможностью выгрузки.

### AC-7: Аудит-лог (User story #13)

* **Given** пользователь создал deck/экспортировал/изменил шаблон,
  **When** событие произошло,
  **Then** оно фиксируется в audit log с: временем, пользователем, ролью, org unit, типом события, deck_id, результатом (success/fail).

### AC-8: Документ-пакет для закупки (User story #15)

* **Given** покупатель открыл раздел “Документы”,
  **When** он скачивает пакет,
  **Then** он получает актуальные версии документов (политики, обработка ПДн/договорные артефакты, подтверждение хостинга), а версии фиксируются.

## 4) Ограничения и зависимости

### 4.1 Ограничения (constraints)

* **Локальность и комплаенс данных:** хостинг и обработка данных должны соответствовать требованиям вуза; по умолчанию предполагается “RF-hosting” и контроль доступа на уровне тенанта.
* **Источник фактов:** система не должна “выдумывать” факты и числа; если данные не предоставлены, маркировать как `NEEDS_SOURCE` и/или задавать уточняющие вопросы.
* **Brand-lock:** пользователь не должен иметь возможности нарушить ключевые параметры брендбука (минимум: шрифты, основные цвета, логотип, отступы/макеты).
* **Экспортная совместимость:** экспортируемый PPTX должен открываться в PowerPoint без критических артефактов; верстка ограничена поддерживаемыми layout’ами.
* **Барьер входа:** демо/пилот должен быть доступен с минимальными барьерами (в идеале — без долгой регистрации), иначе пилот не стартует.

### 4.2 Зависимости (dependencies)

* **SSO/IdP вуза** (SAML/OIDC) и доступ к атрибутам пользователя (роль/орг-единица или маппинг).
* **Шаблоны/брендбук** со стороны вуза (файлы, правила, логотипы).
* **Рендерер PPTX/PDF** (движок генерации слайдов, поддержка шрифтов и макетов).
* **LLM-провайдер / LLM-gateway** (лимиты, стоимость, доступность, соблюдение политики данных).
* (Опционально future) **LMS интеграции** (Moodle и аналоги), офисные пакеты.

## 5) Нефункциональные требования (NFR)

### 5.1 Latency & performance

* **SLO генерации черновика:** целевое время генерации ≤ **30 секунд** для “студенческого” брифа (короткий текст), при показе прогресса.
* **SLO экспорта:** экспорт PPTX/PDF ≤ **20 секунд** для презентации до 15 слайдов.
* **Частичная регенерация:** правка одного слайда должна быть быстрее полной генерации (целевой ориентир: ≤ 10–15 секунд при типовом объёме).
* **Параллелизм:** система должна выдерживать пиковые нагрузки (например, дедлайны) за счёт очередей/лимитов.

### 5.2 Cost

* **Ограничение себестоимости на deck:** система должна поддерживать лимиты на токены/шаги пайплайна и режим “дешёвый черновик → дорогая полировка”.
* **Кэширование:** повторные операции (например, повторный экспорт без изменений) не должны повторно “сжигать” LLM-затраты.

### 5.3 Security

* **Tenant isolation:** данные одного вуза недоступны другому.
* **RBAC:** права по ролям + разграничение по org units (факультет/кафедра).
* **Audit logs:** неизменяемые логи ключевых событий (генерация, экспорт, изменение брендбука, управление правами).
* **Secrets management:** ключи внешних сервисов и токены хранятся в защищённом хранилище; не логируются.

### 5.4 Privacy

* **PII minimization:** не хранить лишние персональные данные; хранить только то, что необходимо для доступа и отчётности.
* **Data retention:** политика хранения брифов/деков/логов (настраиваемая под вуз).
* **Redaction:** при логировании промптов/ответов — маскирование ПДн и чувствительных данных.

### 5.5 Reliability

* **Доступность сервиса:** целевой ориентир MVP — ≥ 99.5% (внутренний показатель).
* **Retry policy:** повторные попытки для временных ошибок (LLM/рендер), с backoff.
* **Idempotency:** операции “generate/export” должны поддерживать идемпотентность по ключу (чтобы избежать дублей при повторах).
* **Graceful degradation:** при недоступности LLM — понятное сообщение и возможность сохранить бриф/попробовать позже.

### 5.6 Observability (для эксплуатации)

* Метрики: BPC, activation rate, exports, time-to-first-deck, ошибки рендера, LLM-ошибки, средняя стоимость на deck.
* Трассировка пайплайна: шаги конвейера (normalize → outline → copy → QA → render), длительности, ретраи.
* Алёрты: рост ошибок экспорта/рост времени генерации/скачок стоимости.

---

## 6) Примечания к MVP (на уровне требований)

* MVP должен обеспечивать **управляемость** (админ-контур, загрузка брендбука, отчётность) и **момент ценности** для конечного пользователя (быстрый branded-экспорт).
* В SRS намеренно закреплён принцип: **“не выдумывать факты”** и маркировать `NEEDS_SOURCE` — это ключ к доверию и пилотам в вузах.
* Частичная регенерация — критична для UX и стоимости: “правим один слайд, не пересобирая всё”.
