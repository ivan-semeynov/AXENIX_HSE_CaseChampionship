Контекст: B2B2C сервис генерации презентаций для российских вузов.
Три роли: Закупщик, Преподаватель, Студент.
Хостинг: только российские серверы (требование 152-ФЗ).


Спроектируй системную архитектуру сервиса. Строго пошагово:


ШАГ 1 — Компоненты системы
Опиши каждый компонент:
— Frontend (веб-интерфейс)
— Backend Orchestrator (управляет пайплайном)
— LLM Gateway (обращения к языковой модели)
— Prompt Library (хранилище промптов)
— Slide Renderer (сборка PPTX/PDF)
— Storage (файлы и база данных)
— Admin Dashboard (дашборд для закупщика)
— Auth Service (SSO + ролевая модель)


Для каждого: роль / с чем взаимодействует / технология с обоснованием


ШАГ 2 — Диаграмма потока данных
Опиши путь запроса от пользователя до готовой презентации:
Пользователь → ... → PPTX/PDF
Текстом со стрелками, каждый шаг на новой строке.


ШАГ 3 — Нефункциональные требования с цифрами
Время генерации, uptime, размер входного файла, 
одновременные пользователи, хранение данных.


ШАГ 4 — Узкие места и как их решаем
Где система может замедлиться или упасть — 
и что делаем в каждом случае.

Документ готов. Что внутри по шагам:
ШАГ 1 — 8 компонентов оформлены карточками с цветной верхней границей. Для каждого: роль, связи с другими компонентами, конкретная технология и обоснование выбора. Ключевые решения — Keycloak для SSO (SAML 2.0 + LDAP из коробки), MinIO для хранения (self-hosted, S3-совместимый, РФ), FastAPI как Orchestrator, python-pptx без зависимости от LibreOffice на продакшне.
ШАГ 2 — Диаграмма потока оформлена как псевдокод в тёмном терминале со стрелками и цветовым выделением ключевых шагов. Отдельно — путь SSO-входа. Важная деталь: Orchestrator возвращает job_idфронтенду немедленно, не блокируя пользователя — вся тяжёлая работа асинхронная.
ШАГ 3 — 10 NFR с цифрами: генерация p50 ≤25 сек, p95 ≤45 сек, uptime 99.5%, пик 2500 concurrent пользователей (5 вузов × 500), хранение 24+36 месяцев, 100% РФ-юрисдикция.
ШАГ 4 — 5 узких мест с цветовой кодировкой критичности: LLM-латентность (решается async-очередью + failover), пиковая нагрузка в день защит (HPA + predictive pre-warming), некорректный JSON от LLM (structured outputs + graceful degradation), вузы без SSO (3 уровня интеграции + ЕСИА в roadmap), зависимость от LLM-провайдера (антикоррупционный слой + multi-provider routing).
