Ниже — **полная декомпозиция** задания трека “Промпт‑инжиниринг” (системный дизайн сервиса генерации презентаций) и **end‑to‑end схемы промптов**, которые закрывают критерии: прозрачность ТЗ для разработчиков, полнота документа, декомпозиция, обоснование формата, “архитектура промпта”, гранулярность.

Я буду исходить из того, что ваш итоговый пакет сдачи состоит из:

1. **ТЗ/Системный дизайн** сервиса (основной документ)
2. **Презентация** как “продукт” по вашему промпту (результат)
3. **Пакет исходных промптов** (promptbook) + краткое обоснование выбора моделей/подхода
4. (опционально) **скрипт выступления** + Q&A

## 0) Как “попасть” в критерии (перевод критериев в deliverables)

### Что жюри хочет увидеть (как артефакты)

**A. Прозрачное ТЗ для разработчиков**

* Чёткая постановка задачи: цели, пользователи, сценарии, границы
* Функциональные требования (FR) + нефункциональные (NFR) + ограничения
* Архитектура (компоненты, потоки данных, API/контракты)
* Риски + меры контроля качества (в т.ч. антигаллюцинации)

**B. Полнота документа**

* Бизнес: ценность, KPI, модель монетизации/внедрения, конкурентные допущения
* Функционал: user stories, acceptance criteria
* Тех: LLM‑оркестрация, хранение данных, безопасность, наблюдаемость, стоимость

**C. Декомпозиция задачи**

* В документе и в промпт‑пайплайне должна быть “лестница” этапов

**D. Обоснование формата постановки**

* Почему вы фиксируете требования в структуре (например: PRD → SRS → Arch → API → Promptbook → Test plan)

**E. “Человеческие компетенции”**

* Покажите, где человек принимает решения: цели, аудитория, тон, структура истории, финальная редактура

**F. Гранулярность промпта**

* Не один “огромный промпт”, а система промптов: intake → план → генерация → проверка → сборка → QA

## 1) Полная декомпозиция end‑to‑end (Work Breakdown Structure)

Ниже — WBS, который можно прямо вставлять в ТЗ как “План работ / Декомпозиция”.

### Этап 1. Discovery / Ввод

1.1. Определить роли пользователей (минимум 2–3):

* Автор презентации (PM/маркетолог/аналитик)
* Ревьюер/руководитель
* Админ/контент‑менеджер (шаблоны, бренд‑гайд)

1.2. Определить форматы входа:

* Текстовый бриф
* Документ (текст/конспект)
* Структура/пункты
* Данные для графиков (таблица/CSV)

1.3. Определить ограничения выхода:

* PPTX/Google Slides/PDF
* Количество слайдов, язык, стиль
* Наличие диаграмм, таблиц, картинок
* Наличие speaker notes

**Артефакт:** “Brief Spec” (структура брифа + правила дефолтов).

---

### Этап 2. Требования (PRD/SRS)

2.1. Бизнес‑цели и KPI:

* Время до черновика, качество, NPS, конверсия в экспорт, удержание
* Ограничение стоимости генерации (например, руб/презентация)

2.2. Функциональные требования (FR):

* Создание презентации по брифу
* Режимы: “Pitch deck / Отчёт / Обучение / Коммерческое предложение”
* Генерация структуры, слайдов, заметок
* Итеративное редактирование: “сократи”, “сделай проще”, “добавь данные”, “в стиле бренда”
* Импорт бренд‑гайда/шаблона
* Генерация графиков по данным
* Экспорт PPTX/PDF, совместная работа, история версий (опционально)

2.3. Нефункциональные требования (NFR):

* Латентность: “черновик ≤ N минут”
* Надёжность/ретраи, лимиты
* Безопасность: PII, секреты, доступы
* Качество: проверка фактов/чисел, отсутствие токсичности, стилевое единство
* Observability: логирование, трассировка промптов, метрики качества

2.4. Ограничения/допущения:

* Какие источники фактов допустимы (веб‑поиск или только входные данные)
* Что делать при недостатке данных: задавать вопросы vs использовать дефолты

**Артефакт:** PRD + SRS (или единый документ “ТЗ”).

---

### Этап 3. Архитектура сервиса

3.1. Компоненты:

* UI/Frontend (веб)
* Backend Orchestrator (workflow)
* LLM Gateway (выбор модели, лимиты, кэш)
* Prompt Library (версии промптов)
* Storage (брифы, результаты, шаблоны)
* Asset service (иконки/картинки/диаграммы)
* Slide Renderer (PPTX/Slides)
* QA/Eval (автопроверки)
* Observability (логи, метрики)

3.2. Потоки данных (основные):

* Intake → План → Сторилайн → Слайды → QA → Render → Export
* Edit loop: правка → частичная регенерация → QA → обновление

3.3. API/контракты между сервисами (минимум 3–5 эндпоинтов)

**Артефакт:** Архитектурная схема + описание потоков.

---

### Этап 4. LLM/Prompt Architecture (ядро трека)

4.1. Пайплайн промптов (многошаговый)

* P0 Normalize Brief
* P1 Clarify Questions (если нужно)
* P2 Storyline Planner
* P3 Slide Outline
* P4 Slide Copywriter (по слайдам)
* P5 Visual Planner (какие визуалы/графики)
* P6 Consistency & QA
* P7 Renderer Spec (JSON для сборки)
* P8 Post‑edit loop prompts

4.2. Техники:

* Role prompting (архитектор/редактор/дизайнер)
* Structured outputs (JSON schema)
* Tree‑of‑thought: 2–3 альтернативы сторилайна → выбор по критериям
* Self‑critique: “сначала сгенерируй, потом проверь по чек‑листу”
* Guardrails: запрет выдумывать цифры, требование пометок [NEEDS SOURCE]

4.3. Human-in-the-loop:

* Точки принятия решения человеком (approve storyboard, approve style tokens)

**Артефакт:** Promptbook + диаграмма пайплайна + правила качества.

---

### Этап 5. Качество и тестирование

* Авто‑чеки: длина буллетов, единый стиль, отсутствие повторов, наличие CTA
* Чек чисел: если число не из входа — пометка
* “Slide rubric”: 10 критериев качества с оценкой 0–2
* Набор тест‑кейсов брифов (3–5 разных жанров)

**Артефакт:** Test Plan + Rubric.

---

### Этап 6. Итоговый продукт для сдачи (презентация)

* Генерируете pitch‑deck про ваш сервис (или любую заданную тему, если так требует хакатон), но **важно**: показать, что результат сделан вашим промпт‑пайплайном.

**Артефакт:** PPTX/PDF + лог промптов (что вы вводили и что получили).

## 2) Схема end‑to‑end промптинга (pipeline как “чертёж”)

Ниже — “сквозной конвейер”. Его можно вставить и в ТЗ как раздел “Алгоритм генерации”.

```
User Brief
  ↓
P0 Brief Normalizer (JSON)
  ↓ (if missing info)
P1 Clarifier (questions or defaults)
  ↓
P2 Storyline Planner (3 options + scoring)
  ↓
P3 Slide Blueprint (slide-by-slide spec)
  ↓
P4 Copywriter (final text + speaker notes)
  ↓
P5 Visual Planner (charts/images/icons, data bindings)
  ↓
P6 QA & Consistency (fixes + issues list)
  ↓
P7 Renderer Spec (PPTX JSON / Google Slides requests)
  ↓
Render Engine → PPTX/PDF
  ↓
Edit loop (targeted regen per slide/component)
```

## 3) “Prompt Cards” — стандарт оформления промптов (чтобы выглядело как инженерия)

Для каждого промпта используйте одинаковую карточку:

**Prompt Card Template**

* ID / Name
* Цель
* Входные переменные
* Выходной формат (строго JSON)
* Правила/ограничения
* Prompt (System/Developer/User)
* Пример входа (опционально)
* Пример выхода (опционально)

Это сильно повышает “прозрачность для разработчиков”.

## 4) Promptbook: набор промптов end‑to‑end (готовые схемы)

Ниже даю **практически готовые тексты**. Вы подставляете свои переменные.

### Общий SYSTEM для всех шагов (ядро качества)

Используйте как “system prompt” для каждого шага (или как Developer message в зависимости от платформы):

```text
Ты — часть конвейера генерации презентаций. Работай строго по задаче шага.
Всегда:
- Пиши на языке {lang}.
- Не выдумывай факты/цифры. Если данных нет во входе — помечай "NEEDS_SOURCE".
- Соблюдай структуру выхода и валидный JSON без комментариев.
- Будь конкретным: каждый пункт должен быть проверяемым.
- Если вход противоречив — перечисли противоречия в поле "risks".
```

# P0 — Brief Normalizer (приведение брифа к структуре)

**Цель:** из любого текста сделать “канонический бриф” + выявить пробелы.
**Вход:** сырой бриф пользователя.
**Выход:** JSON.

```text
Роль: Аналитик требований и редактор брифа.

Задача: Преобразуй сырой бриф в структурированный "PresentationBrief".
1) Извлеки цель, аудиторию, контекст, ключевые сообщения, ограничения, тон, язык, формат, требования к дизайну, источники данных.
2) Определи недостающие поля и сформируй список уточняющих вопросов.
3) Если поле не задано, предложи разумный default и отметь его как "assumed".

Верни строго JSON по схеме:

{
  "brief": {
    "topic": "",
    "goal": "",
    "audience": {"who": "", "level": "", "pain_points": []},
    "context": "",
    "key_messages": [],
    "call_to_action": "",
    "constraints": {
      "slide_count": null,
      "time_minutes": null,
      "language": "{lang}",
      "tone": "",
      "must_include": [],
      "must_avoid": [],
      "brand_guidelines": {"colors": [], "fonts": [], "logo": null}
    },
    "data_inputs": [
      {"type": "text|table|link|none", "description": "", "content": null}
    ],
    "assumptions": []
  },
  "missing_info": [],
  "clarifying_questions": [],
  "risks": []
}

Сырой бриф:
<<<{raw_brief}>>>
```

# P1 — Clarifier (если нужно уточнить, но без лишних вопросов)

**Цель:** задать **минимальный** набор вопросов (3–7), которые максимизируют качество.
**Фишка:** “если можно — предложи дефолт и попроси подтвердить”.

```text
Роль: Продакт-менеджер.

У тебя есть структурированный brief и список missing_info.
Сформируй:
- до 7 уточняющих вопросов (в порядке важности),
- рядом с каждым вопросом предложи дефолт (что будет принято, если пользователь не ответит),
- объясни одним предложением, зачем вопрос.

Верни JSON:
{
  "questions": [
    {"q": "", "default": "", "why_it_matters": ""}
  ]
}

brief:
<<<{brief_json}>>>
```

# P2 — Storyline Planner (3 альтернативы + оценка)

**Цель:** показать “продвинутую технику” (tree-of-thought без раскрытия внутреннего рассуждения) — 3 сторилайна и выбор.

```text
Роль: Стратег по коммуникациям.

Сгенерируй 3 альтернативных структуры презентации (storyline options).
Каждая структура:
- название концепции (например, "Problem–Solution–Proof"),
- список секций,
- логика переходов,
- для кого лучше подходит.

Затем оцени каждую по критериям:
- соответствие цели brief,
- ясность,
- убедительность,
- риск перегруза,
- уникальность.

Выбери одну как "selected" и объясни выбор коротко.

Верни JSON:
{
  "options": [
    {
      "name": "",
      "sections": [{"title": "", "purpose": ""}],
      "flow": "",
      "best_for": "",
      "scores": {
        "goal_fit": 0,
        "clarity": 0,
        "persuasion": 0,
        "overload_risk": 0,
        "uniqueness": 0
      }
    }
  ],
  "selected": {"name": "", "reason": ""}
}

brief:
<<<{brief_json}>>>
```

# P3 — Slide Blueprint (скелет слайдов: назначение, тезисы, визуал)

**Цель:** получить спецификацию “что на каждом слайде”.

```text
Роль: Редактор презентаций уровня консалтинг.

На основе выбранного storyline создай план на {N} слайдов.
Для каждого слайда:
- цель слайда (1 фраза),
- заголовок,
- 3–5 буллетов (короткие, без воды),
- предложение визуала (схема/диаграмма/таблица/иконки),
- что слушатель должен запомнить.

Ограничения:
- Буллет ≤ 12 слов.
- Не используй цифры, если их нет во входных данных (иначе NEEDS_SOURCE).
- Держи стиль и тон из brief.

Верни JSON:
{
  "slides": [
    {
      "slide_no": 1,
      "type": "title|agenda|content|chart|case|summary|cta",
      "objective": "",
      "title": "",
      "bullets": [],
      "visual": {"type": "", "description": "", "data_binding": null},
      "key_takeaway": ""
    }
  ]
}

brief:
<<<{brief_json}>>>
selected_storyline:
<<<{storyline_json}>>>
N: {N}
```

# P6 — QA & Consistency (автокритика + исправления)

**Цель:** продемонстрировать “инженерность”: чек‑лист + фиксы + список проблем.

```text
Роль: Ревьюер качества (QA) для презентаций.

Проверь презентацию по чек-листу:
1) Единый стиль терминов
2) Нет длинных буллетов (>12 слов)
3) Каждый слайд имеет 1 понятный takeaway
4) Нет выдуманных фактов/цифр (если есть — NEEDS_SOURCE)
5) Логика потока: каждый слайд ведёт к следующему
6) Дубли и вода
7) CTA и финальное резюме присутствуют (если требуется)

Сгенерируй:
- список issues,
- исправленную версию slides (если можно исправить без уточнений),
- список "needs_user_input" если без пользователя нельзя.

Верни JSON:
{
  "issues": [{"severity":"low|medium|high","slide_no":null,"problem":"","fix_suggestion":""}],
  "needs_user_input": [{"slide_no": null, "question": "", "why": ""}],
  "fixed_slides": [
    {"slide_no": 1, "title":"", "bullets": [], "speaker_notes": ""}

}
```

## 5) Промпты, чтобы сгенерировать именно “Системный дизайн / ТЗ” (а не только слайды)

Поскольку задание — “разработать системный дизайн сервиса”, вам нужен отдельный пайплайн промптов, который генерирует документ ТЗ.

Ниже — компактный набор: PRD → SRS → Architecture → API → Risks/Test.

### D1 — PRD Generator (бизнес-уровень)

```text
Роль: Product Manager.

Сформируй PRD сервиса генерации презентаций.
Включи:

- Value proposition
- Key use cases
- Success metrics (KPI)
- MVP scope vs Future scope
- Risks & assumptions

Верни Markdown-структуру с заголовками.
Ограничение: 1–2 страницы текста.
```

### D2 — SRS / Functional Spec (FR + user stories + acceptance criteria)

```text
Роль: Системный аналитик.

- Acceptance criteria (Given/When/Then) для ключевых историй
- Ограничения и зависимости
- Нефункциональные требования (latency, cost, security, privacy, reliability)

Верни Markdown.
PRD:
<<<{prd_markdown}>>>
```

## 6) Как упаковать “исходные запросы к инструментам” (чтобы это выглядело как научно)

Сдавайте не просто “простыни”, а **Prompt Pack**:

1. **Prompt Map** (диаграмма пайплайна)
2. **Prompt Cards** (P0–P7, D1–D6)
3. **Run Log** (1 прогон):

* input brief
* output JSON на каждом шаге (можно сокращённо)
* где человек внёс правку

4. **Model rationale** (1 слайд или 1 страница):

* Модель A: лучше в структурировании/архитектуре
* Модель B: лучше в копирайтинге
* Модель C: лучше в длинном контексте
* Принципы: дешевый черновик → дорогая полировка

## 8) Где специально показать “человеческие компетенции” (очень важно)

В вашем ТЗ и презентации явно отметьте 5 точек контроля человеком:

1. Подтверждение цели/аудитории/контекста (после P0–P1)
2. Выбор storyline из 3 вариантов (после P2)
3. Подтверждение tone & brand tokens (перед рендером)
4. Ревью “high severity issues” из QA (после P6)
5. Финальная редактура 1–2 ключевых слайдов (Title + CTA)

Это напрямую закрывает критерий “использование человеческих компетенций”.

## 9) Минимальный “MVP” (если времени мало) — но всё равно выглядит мощно

Если вы ограничены во времени, делайте MVP пайплайн:

* P0 Normalize
* P2 Storyline (2–3 варианта)
* P3 Blueprint
* P4 Copy + notes
* P6 QA
* P7 Render spec

И отдельно D3 + D4 (архитектура + prompt architecture).
Этого достаточно, чтобы выглядеть как системный дизайн, а не “просто текст”.

## 10) Что вы можете сделать прямо сейчас (без уточнений) — последовательность действий команды

1. Возьмите один демонстрационный бриф (про ваш сервис) → прогоните P0 → P2 → P3 → P4 → P6 → P7
2. Параллельно прогоните D1 → D2 → D3 → D4 → D5 → D6 (получите ТЗ)
3. Из D3/D4 сделайте 2–3 ключевых слайда: архитектура, пайплайн промптов, качество/guardrails
4. Сгенерируйте выступление S1 и Q&A S2
5. Соберите “Prompt Pack” (карточки + лог прогона)

---

Если дадите ваш **сырой бриф** (какую именно презентацию вы должны сгенерировать как результат) и/или требования хакатона к формату (сколько слайдов, тематика), я могу:

* собрать **идеальный “brief_json”** под P0,
* предложить **storyline options** под вашу тему,
* и подготовить **готовый план презентации** (slide blueprint) в формате, который удобно рендерить в PPTX.
