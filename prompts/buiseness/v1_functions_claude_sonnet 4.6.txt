Требования из роли преподавателя
Прежде чем собирать полный список, зафиксируем что вытекает из сценария преподавателя.
Из сценария «создаю курс на 16 лекций» — Пакетная генерация: загрузил программу курса → получил набор связанных презентаций с единой структурой, а не один файл — Переиспользование структуры: взял прошлогоднюю лекцию как основу, обновил содержание без переверстки
Из сценария «хочу единый стиль кафедры» — Библиотека шаблонов кафедры: администратор кафедры загружает шаблон, все преподаватели кафедры работают только в нём — Запрет на отклонение от стиля: преподаватель не может случайно изменить корпоративный шрифт или цвет
Из сценария «готовлю методичку и материалы к конференции» — Разные форматы вывода из одного контента: один набор материалов → лекция (много слайдов, тезисы), методичка (структурированный документ), тезисы доклада (5–7 слайдов) — Совместное редактирование: соавтор на кафедре может войти и дополнить презентацию без потери стиля
Из боли «ИИ сгенерирует поверхностно» — Режим «мой текст в основе»: преподаватель загружает свой конспект/статью, сервис структурирует его содержание, не заменяет на generic — Ручная доводка без потери стиля: после генерации можно редактировать текст слайда не сломав шаблон

Итоговая структура v1
20 функций v1, покрывающих все три роли:




Вход и роли         [1–3]   → закупщик убеждается, все входят без трения
Генерация           [4–8]   → студент и преподаватель получают ключевую ценность
Стиль               [9–11]  → вуз получает единый стандарт
Редактирование      [12–13] → доводка без потери стиля
Экспорт             [14–16] → результат работает в реальных условиях защиты
Администрирование   [17–18] → закупщик видит данные для продления
Регуляторика        [19–20] → закупщик может подписать договор




Продукт без любой из этих 20 функций теряет одну из трёх ролей полностью — именно поэтому они в v1.
